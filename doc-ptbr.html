1. Introducao: O Problema Real

Todo projeto bom comeca com um problema real. Neste caso: em uma casa compartilhada na Italia, precisavamos controlar as contas de luz (luce), agua (acqua) e gas. Quem pagou o que? Quanto custou cada mes? Qual a tendencia dos gastos?

A solucao: criar um app web chamado Bollette di Casa ("Contas da Casa" em italiano) que qualquer morador pudesse acessar pelo celular, registrar valores por voz, ver graficos e compartilhar relatorios pelo WhatsApp.

Por que isso importa para voce? Aprender programacao com um projeto real e muito mais eficaz do que tutoriais abstratos. Voce enfrenta problemas verdadeiros (dados duplicados, compatibilidade de navegador, experiencia do usuario) e aprende a resolve-los.

O app esta no ar em: https://guxandobrat.github.io/House-Taxes/

Codigo-fonte: https://github.com/guxandobrat/House-Taxes

2. Arquitetura: Um Unico Arquivo HTML

A Decisao Mais Importante

Optamos por colocar tudo em um unico arquivo index.html: HTML, CSS (dentro de <style>) e JavaScript (dentro de <script>). Sem React. Sem Vue. Sem framework. Sem servidor backend.

Por que um unico arquivo? Simplicidade. Um arquivo HTML pode ser hospedado gratuitamente no GitHub Pages. Sem build step. Sem configuracao de servidor. Sem custos. Basta fazer push e o app esta no ar.

3. Como os Dados Sao Armazenados

IndexedDB: O Banco de Dados do Navegador

Navegadores modernos tem um banco de dados embutido chamado IndexedDB. Diferente do localStorage (que so guarda texto e tem limite de 5MB), o IndexedDB pode armazenar objetos complexos, fazer buscas por indice e lidar com megabytes de dados.

O problema? A API nativa do IndexedDB e horrivel de usar — verbosa, baseada em callbacks, confusa. Por isso usamos o Dexie.js, uma biblioteca que simplifica tudo.

Inicializacao do Banco

// Criando o banco de dados com Dexie.js
const db = new Dexie('BolletteFinal');

// Definindo o schema (estrutura das tabelas)
db.version(1).stores({
  records: '++id, &month'
  // ++id  = chave primaria auto-incremento
  // &month = indice UNICO (nao pode repetir)
});
Por que &month (indice unico)? Queremos apenas UM registro por mes. Se o usuario salvar "luce" em janeiro e depois "gas" em janeiro, os valores devem ir para o mesmo registro. O & garante que o Dexie lance um erro se tentarmos criar dois registros com o mesmo mes — e ai usamos a logica de upsert para mesclar.

Estrutura de um Registro

// Exemplo de um registro no banco
{
  id: 1,                          // auto-gerado
  month: "2024-01",               // chave unica
  luce: 85.50,                    // valor da luz em euros
  luceDate: "2024-01-15",         // data da conta de luz
  acqua: 42.00,                   // valor da agua
  acquaDate: "2024-01-20",        // data da conta de agua
  gas: 67.30,                     // valor do gas
  gasDate: "2024-01-18",          // data da conta de gas
  abitanti: 3,                    // numero de moradores
  createdAt: "2024-01-15T10:30:00Z" // timestamp
}

A Logica de Upsert (Criar ou Atualizar)

Um dos padroes mais importantes do app: quando o usuario salva uma conta, verificamos se ja existe um registro para aquele mes. Se existir, mesclamos os dados. Se nao, criamos um novo.

async function saveRecord(month, type, value, date, abitanti) {
  // 1. Buscar registro existente para esse mes
  const existing = await db.records
    .where('month')
    .equals(month)
    .first();

  if (existing) {
    // 2a. VERIFICAR DUPLICATA: se ja tem valor para essa utilidade
    if (existing[type] !== undefined && existing[type] !== null) {
      alert('Ja existe um valor de ' + type + ' para este mes!');
      return;
    }
    // 2b. UPSERT: mesclar novos dados no registro existente
    await db.records.update(existing.id, {
      [type]: value,
      [type + 'Date']: date,
      abitanti: abitanti
    });
  } else {
    // 3. CRIAR novo registro para o mes
    await db.records.add({
      month: month,
      [type]: value,
      [type + 'Date']: date,
      abitanti: abitanti,
      createdAt: new Date().toISOString()
    });
  }
}

Persistencia de Dados

Navegadores podem decidir apagar dados do IndexedDB quando o espaco em disco esta baixo. Para prevenir isso, usamos a Storage API:

// Pedir ao navegador para NAO apagar nossos dados
if (navigator.storage && navigator.storage.persist) {
  navigator.storage.persist().then(function(granted) {
    if (granted) {
      console.log("Armazenamento persistente concedido!");
    }
  });
}
Atencao: Mesmo com persist(), os dados ficam APENAS naquele navegador. Se o usuario limpar os dados de navegacao, usar modo anonimo ou trocar de aparelho, tudo se perde. Para um app de producao serio, voce precisaria de um backend.

4. Funcionalidades Principais

4.1 Reconhecimento de Voz

O app permite que o usuario dite os valores em vez de digitar. Isso e especialmente util no celular. Usamos a Web Speech API, que esta disponivel nativamente no Chrome.

// Inicializando o reconhecimento de voz
const recognition = new (window.SpeechRecognition
  || window.webkitSpeechRecognition)();

recognition.lang = 'it-IT';  // Italiano como padrao
recognition.continuous = false;
recognition.interimResults = false;

recognition.onresult = function(event) {
  const texto = event.results[0][0].transcript;
  // Extrair numero do texto falado
  const numero = parseFloat(
    texto.replace(/[^\d.,]/g, '').replace(',', '.')
  );
  // Preencher o campo com o valor extraido
  document.getElementById('valor').value = numero;
};
Dica: A Web Speech API so funciona bem no Chrome (desktop e Android). No Safari e Firefox, o suporte e limitado ou inexistente. Sempre tenha um fallback (neste caso, o campo de texto normal).

4.3 Compartilhamento

A Web Share API permite usar o botao nativo de compartilhamento do sistema operacional (WhatsApp, email, Telegram, etc.):

async function shareReport(text) {
  if (navigator.share) {
    await navigator.share({
      title: 'Bollette di Casa',
      text: text  // Relatorio formatado
    });
  }
}

4.4 Tres Idiomas

O app suporta italiano (padrao), espanhol e portugues. A implementacao usa um objeto de traducoes:

const i18n = {
  it: { save: 'Salva', month: 'Mese', ... },
  es: { save: 'Guardar', month: 'Mes', ... },
  pt: { save: 'Salvar', month: 'Mes', ... }

function setLanguage(lang) {
  // Atualizar todos os textos da interface
  document.querySelectorAll('[data-i18n]')
    .forEach(el => {
      el.textContent = i18n[lang][el.dataset.i18n];
    });
}

5. Processo de Desenvolvimento (12+ Iteracoes)

Este app nao nasceu pronto. Ele evoluiu ao longo de mais de 12 versoes, cada uma adicionando funcionalidades ou corrigindo bugs. Isso e normal — software profissional e feito em iteracoes.

v1 - O Comeco

App basico com formulario, reconhecimento de voz, IndexedDB e graficos simples. Tudo funcionava, mas a interface era feia e o codigo desorganizado.

v2 - Redesign Premium

Interface completamente refeita com design profissional. Italiano como idioma padrao. Datas individuais por utilidade (luceDate, acquaDate, gasDate) em vez de uma data unica.

v6 - Compartilhamento

Adicionado compartilhamento via WhatsApp/email com Web Share API. Historico agrupado por mes para visualizacao mais clara.

v8 - Um Registro por Mes

Mudanca arquitetural importante: em vez de um registro por utilidade, agora um unico registro por mes contem luce + acqua + gas. Logica de upsert implementada. Precos exibidos diretamente nos graficos.

v9 - Bloqueio de Duplicatas

Se o usuario tenta salvar "luce" para janeiro e ja existe um valor de luce em janeiro, o app bloqueia e mostra um alerta. Isso previne sobrescrita acidental.

v11 - Limpeza e Filtros

Graficos redundantes removidos. Adicionados botoes de filtro no grafico de Andamento para mostrar/esconder Luce, Acqua e Gas individualmente.

Correcoes de Bugs

Espaco no keypath do IndexedDB causava erro. Indice de mes faltando. Botao de salvar posicionado incorretamente. Cada bug ensinou algo novo.

Por que tantas versoes? Software nunca fica "pronto" na primeira tentativa. Cada versao resolve problemas descobertos pelos usuarios reais. O segredo nao e acertar de primeira, mas iterar rapidamente e melhorar continuamente.

6. Tecnologias Utilizadas

HTML / CSS / JavaScript

As tres linguagens fundamentais da web. Sem frameworks. O HTML define a estrutura, o CSS o visual e o JS a logica. Dominar essas tres e a base de tudo.

Dexie.js

Wrapper para IndexedDB que transforma uma API complicada em chamadas simples com Promises e async/await. Tamanho minimo (~45KB). Por que: a API nativa do IndexedDB exige muito codigo boilerplate.

Chart.js

Biblioteca de graficos responsivos baseada em Canvas. Facil de usar, bonita por padrao e com ecossistema de plugins. Por que: criar graficos SVG manualmente seria extremamente trabalhoso.

chartjs-plugin-datalabels

Plugin para Chart.js que exibe valores diretamente nas barras/linhas dos graficos. Por que: sem ele, o usuario precisa passar o mouse sobre cada ponto para ver o valor.

Web Speech API

API nativa do navegador para reconhecimento de voz. Sem custo, sem API key. Limitacao: funciona bem apenas no Chrome. Por que: entrada por voz e mais pratica no celular do que digitar numeros.

Web Share API

API nativa para compartilhamento. Abre o menu de compartilhamento do sistema operacional (WhatsApp, email, etc.). Por que: integracao nativa e melhor que links manuais de "compartilhar no WhatsApp".

GitHub Pages

Hospedagem gratuita para sites estaticos diretamente de um repositorio GitHub. HTTPS incluido. Por que: custo zero e deploy automatico a cada push.

CDN (Content Delivery Network)

Bibliotecas externas carregadas via CDN (jsDelivr/cdnjs) em vez de incluir os arquivos. Por que: menos codigo no repositorio e cache compartilhado entre sites.

7. Como o GitHub Pages Funciona

GitHub Pages e um servico gratuito do GitHub que transforma um repositorio em um site web. Veja como:

Passo a Passo

  1. Criar um repositorio no GitHub (ex: House-Taxes)
  2. Adicionar um index.html na raiz do repositorio
  3. Ativar GitHub Pages: Settings → Pages → Source: "Deploy from branch" → Branch: main
  4. Aguardar o deploy (geralmente 1-2 minutos)
  5. Acessar: https://SEU-USUARIO.github.io/NOME-DO-REPO/

Vantagens

  • Custo zero — totalmente gratuito para repositorios publicos
  • HTTPS automatico — certificado SSL gratis (necessario para Web Speech API e Web Share API)
  • Deploy automatico — cada git push atualiza o site
  • Uptime excelente — infraestrutura do GitHub
Limitacoes: Apenas sites estaticos (HTML/CSS/JS). Nao roda PHP, Python, Node.js no servidor. Limite de 1GB para o repositorio e 100GB de banda por mes (mais que suficiente para um app pessoal).

Por que HTTPS e Importante

Varias APIs modernas do navegador exigem conexao HTTPS:

  • navigator.storage.persist() — persistencia de dados
  • SpeechRecognition — reconhecimento de voz
  • navigator.share() — compartilhamento nativo

Sem HTTPS, essas funcionalidades simplesmente nao funcionam. GitHub Pages fornece HTTPS automaticamente.

8. Licoes Aprendidas

8.1 Comece Simples, Melhore Depois

A v1 era feia e basica, mas funcionava. Isso e mais valioso do que um design perfeito que nunca fica pronto. Lance a versao minima viavel (MVP) e itere.

8.2 Bugs Sao Professores

Um espaco no keypath do IndexedDB ("++ id" em vez de "++id") causou horas de debug. Um indice de mes faltando fazia buscas falharem silenciosamente. Cada bug ensinou algo que nenhum tutorial ensinaria.

Dica: Quando encontrar um bug, antes de corrigi-lo, entenda por que ele aconteceu. Isso te faz um programador melhor do que simplesmente copiar uma solucao do Stack Overflow.

8.3 APIs Nativas Sao Poderosas

Antes de instalar uma biblioteca, verifique se o navegador ja tem uma API para o que voce precisa. Web Speech API, Web Share API, IndexedDB, Service Workers — os navegadores modernos fazem muito mais do que voce imagina.

8.4 Dados Sao o Coracao do App

O design mudou varias vezes. Funcionalidades foram adicionadas e removidas. Mas a modelagem dos dados (um registro por mes com campos para cada utilidade) foi a decisao que mais impactou a arquitetura. Invista tempo pensando na estrutura dos dados antes de comecar a codar.

8.5 Custo Zero e Possivel

Este app roda 24/7, com HTTPS, para qualquer numero de usuarios, sem custar um unico centavo. A combinacao GitHub Pages + IndexedDB + CDNs gratuitas torna possivel criar e publicar software real sem cartao de credito.

8.6 Pense no Usuario

Funcionalidades como reconhecimento de voz e compartilhamento nativo existem porque pensamos em como o usuario usaria o app (no celular, na cozinha, querendo enviar o relatorio para o colega de casa). Tecnologia pela tecnologia nao tem valor — tecnologia que resolve um problema, sim.

Resumo Final

O que construimos: Um app web completo para rastrear contas domesticas, com voz, graficos e compartilhamento.

Como: Um unico arquivo HTML com CSS e JavaScript inline, usando bibliotecas via CDN.

Onde: GitHub Pages (gratuito, HTTPS, deploy automatico).

Dados: IndexedDB via Dexie.js, tudo no navegador do usuario.

Maior licao: Voce nao precisa de ferramentas caras ou frameworks complexos para criar software util. Precisa de um problema real, as tecnologias certas e vontade de iterar.
