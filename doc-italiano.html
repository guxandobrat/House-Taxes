1. Introduzione: Il Problema Reale

Ogni buon progetto nasce da un problema reale. In questo caso: in una casa condivisa in Italia, avevamo bisogno di controllare le bollette di luce, acqua e gas. Chi ha pagato cosa? Quanto e costato ogni mese? Qual e la tendenza delle spese?

La soluzione: creare un'app web chiamata Bollette di Casa che qualsiasi coinquilino potesse usare dal cellulare, registrare valori con la voce, vedere grafici e condividere report via WhatsApp.

Perche questo e importante per te? Imparare a programmare con un progetto reale e molto piu efficace dei tutorial astratti. Affronti problemi veri (dati duplicati, compatibilita del browser, esperienza utente) e impari a risolverli.

2. Architettura: Un Solo File HTML

La Decisione Piu Importante

Abbiamo scelto di mettere tutto in un solo file index.html: HTML, CSS (dentro <style>) e JavaScript (dentro <script>). Niente React. Niente Vue. Nessun framework. Nessun server backend.

Perche un solo file? Semplicita. Un file HTML puo essere ospitato gratuitamente su GitHub Pages. Nessun build step. Nessuna configurazione server. Nessun costo. Basta fare push e l'app e online.

3. Come Vengono Salvati i Dati

IndexedDB: Il Database del Browser

I browser moderni hanno un database integrato chiamato IndexedDB. A differenza di localStorage (che salva solo testo e ha un limite di 5MB), IndexedDB puo memorizzare oggetti complessi, fare ricerche per indice e gestire megabyte di dati.

Il problema? L'API nativa di IndexedDB e terribile da usare — verbosa, basata su callback, confusa. Per questo usiamo Dexie.js, una libreria che semplifica tutto.

Inizializzazione del Database

// Creazione del database con Dexie.js
const db = new Dexie('BolletteFinal');

// Definizione dello schema (struttura delle tabelle)
db.version(1).stores({
  records: '++id, &month'
  // ++id  = chiave primaria auto-incremento
  // &month = indice UNICO (non puo ripetersi)
});
Perche &month (indice unico)? Vogliamo un solo record per mese. Se l'utente salva "luce" a gennaio e poi "gas" a gennaio, i valori devono andare nello stesso record. Il & garantisce che Dexie lanci un errore se si tenta di creare due record con lo stesso mese — e noi usiamo la logica di upsert per fondere i dati.

Struttura di un Record

// Esempio di un record nel database
{
  id: 1,                          // auto-generato
  month: "2024-01",               // chiave unica
  luce: 85.50,                    // importo della luce in euro
  luceDate: "2024-01-15",         // data della bolletta luce
  acqua: 42.00,                   // importo dell'acqua
  acquaDate: "2024-01-20",        // data della bolletta acqua
  gas: 67.30,                     // importo del gas
  gasDate: "2024-01-18",          // data della bolletta gas
  abitanti: 3,                    // numero di coinquilini
  createdAt: "2024-01-15T10:30:00Z" // timestamp
}

La Logica di Upsert (Crea o Aggiorna)

Uno dei pattern piu importanti dell'app: quando l'utente salva una bolletta, controlliamo se esiste gia un record per quel mese. Se esiste, fondiamo i dati. Se no, ne creiamo uno nuovo.

async function saveRecord(month, type, value, date, abitanti) {
  // 1. Cercare un record esistente per questo mese
  const existing = await db.records
    .where('month')
    .equals(month)
    .first();

  if (existing) {
    // 2a. VERIFICA DUPLICATO: se ha gia un valore per questa utenza
    if (existing[type] !== undefined && existing[type] !== null) {
      alert('Esiste gia un valore di ' + type + ' per questo mese!');
      return;
    }
    // 2b. UPSERT: fondere i nuovi dati nel record esistente
    await db.records.update(existing.id, {
      [type]: value,
      [type + 'Date']: date,
      abitanti: abitanti
    });
  } else {
    // 3. CREARE nuovo record per il mese
    await db.records.add({
      month: month,
      [type]: value,
      [type + 'Date']: date,
      abitanti: abitanti,
      createdAt: new Date().toISOString()
    });
  }
}

Persistenza dei Dati

I browser possono decidere di cancellare i dati di IndexedDB quando lo spazio su disco e basso. Per prevenirlo, usiamo la Storage API:

// Chiedere al browser di NON cancellare i nostri dati
if (navigator.storage && navigator.storage.persist) {
  navigator.storage.persist().then(function(granted) {
    if (granted) {
      console.log("Archiviazione persistente concessa!");
    }
  });
}
Attenzione: Anche con persist(), i dati rimangono SOLO in quel browser. Se l'utente cancella i dati di navigazione, usa la modalita anonima o cambia dispositivo, tutto si perde. Per un'app di produzione seria, avresti bisogno di un backend.

4. Funzionalita Principali

4.1 Riconoscimento Vocale

L'app permette all'utente di dettare i valori invece di digitarli. Questo e particolarmente utile sul cellulare. Usiamo la Web Speech API, disponibile nativamente in Chrome.

// Inizializzazione del riconoscimento vocale
const recognition = new (window.SpeechRecognition
  || window.webkitSpeechRecognition)();

recognition.lang = 'it-IT';  // Italiano come default
recognition.continuous = false;
recognition.interimResults = false;

recognition.onresult = function(event) {
  const testo = event.results[0][0].transcript;
  // Estrarre il numero dal testo parlato
  const numero = parseFloat(
    testo.replace(/[^\d.,]/g, '').replace(',', '.')
  );
  // Compilare il campo con il valore estratto
  document.getElementById('valore').value = numero;
};
Consiglio: La Web Speech API funziona bene solo su Chrome (desktop e Android). Su Safari e Firefox il supporto e limitato o inesistente. Prevedi sempre un fallback (in questo caso, il campo di testo normale).

4.3 Condivisione

La Web Share API permette di usare il pulsante nativo di condivisione del sistema operativo (WhatsApp, email, Telegram, ecc.):

async function shareReport(text) {
  if (navigator.share) {
    await navigator.share({
      title: 'Bollette di Casa',
      text: text  // Report formattato
    });
  }
}

4.4 Tre Lingue

L'app supporta italiano (default), spagnolo e portoghese. L'implementazione usa un oggetto di traduzioni:

const i18n = {
  it: { save: 'Salva', month: 'Mese', ... },
  es: { save: 'Guardar', month: 'Mes', ... },
  pt: { save: 'Salvar', month: 'Mes', ... }

function setLanguage(lang) {
  // Aggiornare tutti i testi dell'interfaccia
  document.querySelectorAll('[data-i18n]')
    .forEach(el => {
      el.textContent = i18n[lang][el.dataset.i18n];
    });
}

5. Processo di Sviluppo (12+ Iterazioni)

Quest'app non e nata pronta. Si e evoluta nel corso di piu di 12 versioni, ognuna aggiungendo funzionalita o correggendo bug. Questo e normale — il software professionale viene fatto a iterazioni.

v1 - L'Inizio

App base con form, riconoscimento vocale, IndexedDB e grafici semplici. Tutto funzionava, ma l'interfaccia era brutta e il codice disorganizzato.

v2 - Redesign Premium

Interfaccia completamente rifatta con design professionale. Italiano come lingua predefinita. Date individuali per utenza (luceDate, acquaDate, gasDate) invece di una data unica.

v6 - Condivisione

Aggiunta condivisione via WhatsApp/email con Web Share API. Storico raggruppato per mese per una visualizzazione piu chiara.

v8 - Un Record per Mese

Cambiamento architetturale importante: invece di un record per utenza, ora un unico record per mese contiene luce + acqua + gas. Logica di upsert implementata. Prezzi visualizzati direttamente sui grafici.

v9 - Blocco Duplicati

Se l'utente tenta di salvare "luce" per gennaio e esiste gia un valore di luce per gennaio, l'app blocca e mostra un avviso. Questo previene la sovrascrittura accidentale.

v11 - Pulizia e Filtri

Grafici ridondanti rimossi. Aggiunti pulsanti filtro nel grafico Andamento per mostrare/nascondere Luce, Acqua e Gas singolarmente.

Correzioni Bug

Spazio nel keypath di IndexedDB causava errore. Indice del mese mancante. Pulsante Salva posizionato in modo errato. Ogni bug ha insegnato qualcosa di nuovo.

Perche cosi tante versioni? Il software non e mai "pronto" al primo tentativo. Ogni versione risolve problemi scoperti dagli utenti reali. Il segreto non e fare tutto giusto la prima volta, ma iterare rapidamente e migliorare continuamente.

6. Tecnologie Utilizzate

HTML / CSS / JavaScript

Le tre linguaggi fondamentali del web. Senza framework. L'HTML definisce la struttura, il CSS l'aspetto visivo e il JS la logica. Padroneggiare queste tre e la base di tutto.

Dexie.js

Wrapper per IndexedDB che trasforma un'API complicata in chiamate semplici con Promises e async/await. Dimensione minima (~45KB). Perche: l'API nativa di IndexedDB richiede troppo codice boilerplate.

Chart.js

Libreria di grafici responsive basata su Canvas. Facile da usare, bella di default e con ecosistema di plugin. Perche: creare grafici SVG manualmente sarebbe estremamente laborioso.

chartjs-plugin-datalabels

Plugin per Chart.js che visualizza i valori direttamente sulle barre/linee dei grafici. Perche: senza di esso, l'utente deve passare il mouse su ogni punto per vedere il valore.

Web Speech API

API nativa del browser per il riconoscimento vocale. Senza costo, senza API key. Limitazione: funziona bene solo su Chrome. Perche: l'input vocale e piu pratico sul cellulare che digitare numeri.

Web Share API

API nativa per la condivisione. Apre il menu di condivisione del sistema operativo (WhatsApp, email, ecc.). Perche: l'integrazione nativa e meglio dei link manuali "condividi su WhatsApp".

GitHub Pages

Hosting gratuito per siti statici direttamente da un repository GitHub. HTTPS incluso. Perche: costo zero e deploy automatico ad ogni push.

CDN (Content Delivery Network)

Librerie esterne caricate via CDN (jsDelivr/cdnjs) invece di includere i file. Perche: meno codice nel repository e cache condivisa tra siti.

7. Come Funziona GitHub Pages

GitHub Pages e un servizio gratuito di GitHub che trasforma un repository in un sito web. Ecco come:

Passo dopo Passo

  1. Creare un repository su GitHub (es: House-Taxes)
  2. Aggiungere un index.html nella root del repository
  3. Attivare GitHub Pages: Settings → Pages → Source: "Deploy from branch" → Branch: main
  4. Aspettare il deploy (di solito 1-2 minuti)
  5. Accedere: https://TUO-UTENTE.github.io/NOME-REPO/
Il Tuo Computer GitHub GitHub Pages CDN +-----------+ +-------------+ +-------------------+ | index.html| ---> | Repository | ---> | Sito pubblicato | | (git push)| | House-Taxes | | HTTPS automatico | +-----------+ +-------------+ +-------------------+ | | | Modificare git push Gli utenti accedono il codice = deploy! dal cellulare Figura 3: Flusso di deploy con GitHub Pages

Vantaggi

  • Costo zero — totalmente gratuito per repository pubblici
  • HTTPS automatico — certificato SSL gratis (necessario per Web Speech API e Web Share API)
  • Deploy automatico — ogni git push aggiorna il sito
  • Uptime eccellente — infrastruttura di GitHub
Limitazioni: Solo siti statici (HTML/CSS/JS). Non esegue PHP, Python, Node.js sul server. Limite di 1GB per il repository e 100GB di banda al mese (piu che sufficiente per un'app personale).

Perche HTTPS e Importante

Diverse API moderne del browser richiedono connessione HTTPS:

  • navigator.storage.persist() — persistenza dei dati
  • SpeechRecognition — riconoscimento vocale
  • navigator.share() — condivisione nativa

Senza HTTPS, queste funzionalita semplicemente non funzionano. GitHub Pages fornisce HTTPS automaticamente.

8. Lezioni Imparate

8.1 Inizia Semplice, Migliora Dopo

La v1 era brutta e basica, ma funzionava. Questo ha piu valore di un design perfetto che non viene mai completato. Lancia la versione minima funzionante (MVP) e itera.

8.2 I Bug Sono Maestri

Uno spazio nel keypath di IndexedDB ("++ id" invece di "++id") ha causato ore di debug. Un indice di mese mancante faceva fallire le ricerche silenziosamente. Ogni bug ha insegnato qualcosa che nessun tutorial avrebbe insegnato.

Consiglio: Quando trovi un bug, prima di correggerlo, capisci perche e successo. Questo ti rende un programmatore migliore che semplicemente copiare una soluzione da Stack Overflow.

8.3 Le API Native Sono Potenti

Prima di installare una libreria, controlla se il browser ha gia un'API per cio che ti serve. Web Speech API, Web Share API, IndexedDB, Service Workers — i browser moderni fanno molto piu di quanto immagini.

8.4 I Dati Sono il Cuore dell'App

Il design e cambiato diverse volte. Funzionalita sono state aggiunte e rimosse. Ma la modellazione dei dati (un record per mese con campi per ogni utenza) e stata la decisione che ha impattato di piu l'architettura. Investi tempo a pensare alla struttura dei dati prima di iniziare a scrivere codice.

8.5 Costo Zero e Possibile

Quest'app gira 24/7, con HTTPS, per qualsiasi numero di utenti, senza costare un solo centesimo. La combinazione GitHub Pages + IndexedDB + CDN gratuite rende possibile creare e pubblicare software reale senza carta di credito.

8.6 Pensa all'Utente

Funzionalita come riconoscimento vocale e condivisione nativa esistono perche abbiamo pensato a come l'utente userebbe l'app (sul cellulare, in cucina, volendo inviare il report al coinquilino). La tecnologia per la tecnologia non ha valore — la tecnologia che risolve un problema, si.

Riepilogo Finale

Cosa abbiamo costruito: Un'app web completa per tracciare le bollette domestiche, con voce, grafici e condivisione.

Come: Un solo file HTML con CSS e JavaScript inline, usando librerie via CDN.

Dove: GitHub Pages (gratuito, HTTPS, deploy automatico).

Dati: IndexedDB via Dexie.js, tutto nel browser dell'utente.

Lezione piu grande: Non servono strumenti costosi o framework complessi per creare software utile. Serve un problema reale, le tecnologie giuste e la voglia di iterare.
